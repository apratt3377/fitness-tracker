== Building Block View

Fitness Tracker is a microservices based application. 
Start with building-blocks-starter
The SPA is a React frontend
Focus on internal microservices

=== Whitebox Overall System

_**<Overview Diagram>**_

Motivation::

I used a microservices architecture to implement this application. I wanted to explore how to design a system with multiple microservices that communicate with each other asynchronously. Though this could have been a monolithic application, I wanted to explore the challenges and benefits of decoupling services. These services are primarily written in java as I wanted to evaluate it's new features since I last extensively worked with it.

Contained Building Blocks::

[cols="1,2" options="header"]
|===
| **Name** | **Responsibility**
| React SPA | React Frontend for users and admins to interact with the system
| API Gateway | Routes requests to appropriate microservices
| Auth Service | Handles user authentication and authorization. Stores session data.
| User Service | Manages user data and profiles. Persists user data.
| Workout Management Service | Core logic for managing workouts. Persists workout data of users.
| Notification Service | Notifies users about AI-generated recommendations and workout summaries. Persists AI recommendation messages.
| Analytics Service | Generates AI analysis of completed workouts using Gemini API
| Message Broker | SNS + SQS fanout for event driven communication between microservices
|===

All of the above blackboxes are standalone microservices except for the message broker which is a shared infrastructure component.

Then for each black box describe intent/responsibility, interfaces, files?


Important Interfaces::
[cols="1,2" options="header"]
|===
| **Name** | **Responsibility**
| Public API Gateway | The primary REST entry point for React SPA. Handles routing to User, Workout, and Notification services.
| Auth/Identity API | Internal REST interface used by API Gateway to validate JWTs and auth service to verify credentials.
| Workout Stream (async) | Message-based interface for publishing workout events to other microservices.
| Analytics Stream | Consumes events from Workout Stream to generate AI analysis
| Notification Events | Internal event based interface to trigger real time push notifications to users
| Gemini AI Integration | Outbound REST interface to Google Gemini API
| Persistence to DB| JDBC interface between microservices and their respective databases
|===

==== React SPA (Blackbox)

.Responsibility
The React SPA provides the user interface for both users and admins to interact with the fitness-tracker system. It is a simple, standard React app.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| REST interface | Communicates with the API Gateway
| SSE interface | Receives real time notifications from Notification Service
|===

.Files
The React SPA code is located in the `fitness-tracker-frontend` directory.

==== API Gateway (Blackbox)

.Responsibility
The API Gateway serves as the single entry point for all client requests. It handles request routing, authentication and load balancing. It utilizes Kubernetes discovery service to resolve microserivce instances.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| External | Exposes REST HTTP endpoints for clients to interact with
| Internal | Routes requests to appropriate microservices using kubernetes discovery service
|===

==== Auth Service (Blackbox)

.Responsibility
Manages security lifecycle of user session. It performs credential verification, issues signed JWTs and manages session state.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| Inbound | REST API for login/logout
| Outbound (Sync) | REST API call to User Service for identity data
| Outbound (State) | Redis for session persistence
|===

==== User Service (Blackbox)

.Responsibility
User service manages the lifecycle of a user (registration, profile, deactivation). It serves as source of truth for Auth Service to authenticate and authorize users.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| REST Interface + 
`/api/v1/users` | Public REST API for profile management and user registration
| Internal REST Interface +
`/internal/v1/identity` | Internal interface used by Auth Service to retrieve hashed credentials
| User DB | Private PostgreSQL layer for persisting users
|===

==== Workout Management Service (Blackbox)

.Responsibility
Workout Management Service is the core service for managing the lifecycle of user workouts, supporting full CRUD operations. It acts as a producer for the asynchronous data stream, publishing completed workout data to be processed downstream by analytics services.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| Public REST Interface + 
`/api/v1/workouts` | Public REST API for users to submit workout data and view workout history 
| Workout Event Stream +
`Workout.Domain.Events` | Async Messaging via SNS/SQS. Publishes events for consumption by Analytics service
| Fitness DB | Private PostgreSQL layer for persisting workout data
|===

==== Notification Service (Blackbox)

.Responsibility
The Notification Service serves as a stateful aggregator for workout insights and real time delivery hub for the React frontend. It persists AI-generated analysis results and 
managed open HTTP streams (SSE) to alert the React frontend of new data.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| SSE Interface + 
`/api/v1/notifications/stream` | Persistent connection used to send signals of new notifications to frontend
| Public REST interface +
`/api/v1/notifications` | Public REST API for retrieving and managing notifications
| Notification DB | Private MongoDB layer to persist notification data
| SQS Consumer | Async consumer of Analysis results and workout lifecycle (deletion) events
|===

==== Analytics Service (Blackbox)

.Responsibility
The Analytics service processes completed workout data to generate user feedback via AI. It sends workout data with a prompt to Gemini API for processing and publishes resulting insights to the notification stream

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| Workout Event Stream Consumer + 
`Workout.Domain.Events` | Consumes events to trigger AI analysis workflow
| AI Analysis Client | HTTPS requests to Google API (Gemini) 
| Notification Stream | Publishes final AI-generated insights to notification service
|===

==== Message Broker (Blackbox)

.Responsibility
The Message Broker provides the asynchronous communication backbone for the system. It's resonsible for reliable distribution of events to downstream services. Ensures that workout processing, data cleanup, and notification delivery are decoupled and handled independently.

.Interfaces
[cols="1,2" options="header"]
|===
| **Name** | **Description**
| Analytics Queue + 
`Filter: type=process` | Receives workout completion events intended for AI processing
| Cleanup Queue +
`Filter: type=delete` | Receives workout lifecycle events to ensure data is purged properly
| Notification Queue | Point to point buffere between Analytics and Notification services
|===

==== White Box: Message Broker

<put in sqs image>

.Internal Components
[cols="1,2" options="header"]
|===
| **Component** | **Responsibility**
| Workout.Domain.Events (SNS) | Central entrypoint. Evaluates message attributes to decide which downstream queues receives the message
| Analytics Queue (SQS) | Bufferes events tagged with `process` filter for Analytics Service
| Cleanup Queue (SQS) | Buffers events tagged with `delete` filter to trigger data removal in Notification service
| Notification Queue (SQS) | Buffer between Analytics Service and Notification Service
| Dead Letter Queues | Store messages that have failed processing after multiple retries, allowing for manual recovery without blocking main queues
|===
