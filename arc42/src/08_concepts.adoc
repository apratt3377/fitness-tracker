== Cross-cutting Concepts

=== Domain Models

The system follows a Database-per-Service pattern to ensure service autonomy and high availability. In this system, the domain model is partitioned across three specialized data stores. To maintain consistency and traceability, the system relies on UUIDs

The `user_id` generated by the User Service acts as the primary ownership key across all services.

The `workout_id` connects a physical workout session to its AI-generated analysis and eventual cleanup.

.Data Model Mapping
[cols="1,2" options="header"]
|===
| Domain Entity | Storage Type | Primary Role | Lifecycle Policy
| Identity & Profile | Postgres (User DB) | Authentication and Authorization | Permanent until account deletion
| Workout Session | Postgres (Fitness DB) | Transactional Record | Soft delete (deleted_at); Hard Delete via Cleanup queue
| AI Insights | MongoDB (Notification DB) | Read-Optimized Mailbox | TTL Index
|===

==== Fitness DB (Postgres)
Database for the Workout Management Service

image::workout-entity-diagram.png[]

.Tables
[cols="1,2" options="header"]
|===
| Name | Description
| Workout | Represents a single training session for a user. Specifies the name, description, date and status. Also has auditing columns for created and updated.
| WorkoutExercise | Links a `Workout` to specific `Exercises`. It tracks performance data for that session, including number of sets, reps and weight used. 
| Exercises | A global catalog of available exercises to include in a workout. Stores name, target muscle group and difficulty level.
|===

.Constraints
[cols="1,2,4" options="header"]
|===
| Table | Constraint | Purpose
| Workout | `NOT NULL` on `user_id` and `status` | Ensures every workout is owned by a user and has a lifecycle state
| Workout | `CHECK` on `status` | Restricts status column to values of PLANNED, COMPLETED
| Exercise | `NOT NULL` on `name` | every exercise must be identifiable
| Exercise | `UNIQUE` on `name` | prevents duplicate entries in exercise catalog
| WorkoutExercise | `FOREIGN KEY` (`workout_id`) `ON DELETE CASCADE` | Deletes the entry in this table if parent Workout is deleted.
| WorkoutExercise | `FOREIGN KEY` (exercise_id) `ON DELETE RESTRICT` | Prevents an exercise from being deleted if its being used in any user workouts.
| WorkoutExercise | `CHECK` on `sets`, `reps`, `weight` | Prevent junk data.
|===

==== User DB (Postgres)
Database for the User Service

User DB has one table representing users of the system. Each record represents a unique user, containing their authentication credentials (username and hash of password), user profile and roles.

|===

==== Notification DB (Mongodb)
The Notification DB is a stateful buffer between the asynchronous AI processing and the Client. It is a NoSQL document store designed to handle AI-generated insights, which could come in different formats depending on the prompt. The database is optimized for flexible schemas and automated data expiration.

This is the primary collection used by the service:

| Field | Description

| workout_id | Correlation Id that links the insight to a specific workout session.
| type | Category of insight
| payload | raw AI output (text, structured JSON, etc)
| status | READ or UNREAD
| createdAt | point of creation used for TTL
|===

==== Auth (Redis)

To support the hybrid JWT-Redis security strategy, Redis is used as a high performance session registry. 

.Implementation
The registry is implemented as a manual key-value store using Spring Data Redis, which ensures a language-agnostic data format. Time to Live is used and matches the JWT `exp` claim. A logout triggers an immediate deletion of the `jti` key in Redis.

Data Structure:

* *key:* `auth:session:{jti}` where `{jti}` is the unique JWT ID claim
* *value:* a JSON String representing minimal session context:
 `{"userId": "123", "roles": ["USER"], "issuedAt": ...}`

=== Persistence

Access to the databases is through Spring Data JPA.

=== Security & Session Handling
This system uses a hybrid JWT-Redis approach.

* *Authentication:* The Authorization Service validates login credentials against the PostgreSQL User DB. It generates a JWT containing a unique `jti` claim and stores the `jti` in Redis using a key value pair where key: `auth:session:{jti}` and value: JSON object representing user session (ie `{"userId": "123", "roles": ["USER"]}`). It returns the JWT back to the client.
* *Session Management:* The API Gateway validates the JWT and then extracts the `jti` and checks that it exists in Redis. If the session is revoked or expired, the request is rejected.
* *Identity Propagation:* To decouple internal services from Auth provider, the Gateway translates the JWT into internal headers. It injects `X-User-ID` and `X-User-Roles` headers into downstream requests. Internal microservices are configured to trust this header from the Gateway identifying the user. Potentially `X-User-Roles` header will be used to pass RBAC roles to be verified by microservices.
* *Authorization (RBAC):* Access control is enforced at the API Gateway based on JWT claims. Microservices may use method-level security with `@PreAuthorize` annotation.

=== User Interface
The user interface for {project-name} is a Single Page Application written in React. Both Admin and Users will utilize this UI. The UI will require login for user authentication via the API Gateway.

=== Communication and Messaging
The system uses LocalStack to simulate AWS SNS and SQS in a local Docker environment

* *Aysnc Processing:* Long-running AI tasks are offloaded to SQS
* *Correlation IDs:* To ensure traceability across async flow

=== Resilience and Fault Tolerance

* *Fault Tolerance (DLQs):* All SQS queues are configured with a Dead Letter Queue. If a service fails to process after 3 retries, the message is moved to the DLQ for manual inspection or later retry. Ordering of messages is not important in this project.

* *Scalability:* All Services are stateless. All session state is externalied to Redis.

=== Error Handling and Validity checks

* *Validity Checks:* Handled at Controllers with JSR-303/Bean Validation for immediate Request validation. Also handled at Entity level with database constraints to ensure final data integrity.

* *Global Error Handling:* Each Spring Boot service uses @ControllerAdvice to standardize handling exceptions throughout the system. Allows the Frontend to receive consistent error formats.

=== Observability (Loggin and Tracing)
All services log to STDOUT in standardized format. Use a traceId for distributed tracing to link logs across different microservices.

=== Configurability and Build Management

* *Java/Spring:* configuration managed via application.yml
* *Python:* Analytics service uses .env files and environment variables
* *Build:* Maven handles Java ecosystem. Docker is common packaging format for all services
* *Testability:* Target of 70% unit test coverage using JUnit5 and Mockito is required for all backend services.
